## 1. 实现摘要

在 `uni/` 下实现了最小异构调度层和四卡并行 FP64 基线验证。

### 新增文件

```
uni/
├── src/scheduler/
│   ├── devices.py          # 设备发现 (lspci + sysfs)
│   ├── phi.py              # Phi 编译(podman ICC) + 运行(micnativeloadex)
│   └── ve.py               # VE 编译(ncc) + 运行(ve_exec)
├── src/kernels/
│   ├── phi/peak_fp64.c     # 复用 intel_phi/tests/perf/phi_peak_fp64.c
│   └── ve/peak_fp64.c      # 新写: 数组FMA循环, 触发编译器自动向量化
├── scripts/run_verify.py   # 入口脚本: 发现→编译→并行执行→报告
└── docs/
    ├── plan/20260603_090000_v1_verification_case.md  # 规划文档
    └── impl/20260603_090000_v1_verification.md       # 本文件
```

## 2. 硬件环境确认

| 设备 | PCIe地址 | NUMA | 在线 |
|------|---------|------|------|
| Phi 7120P | 5e:00.0 | 0 | ✅ |
| VE0 | 3b:00.0 | 0 | ✅ |
| VE1 | af:00.0 | 1 | ✅ |
| VE2 | d8:00.0 | 1 | ✅ |

关键发现:
- VE 的 `ve_exec -N` 从 1 开始（非 0），映射为 ve0→-N1, ve1→-N2, ve2→-N3
- Phi 通过 `micnativeloadex` 运行需设置 `SINK_LD_LIBRARY_PATH` 指向 `intel_phi/icc_mic_libs/`
- Phi 编译需通过 podman 容器 `centos7-phi-dev`（需先 `podman start`）
- Python 3.11 可用（系统默认 3.6 不支持 dataclasses）

## 3. 验证结果

```
设备       类型     NUMA   GFLOPS       用时         状态
phi0     phi    0      577.6        0.45s      ✅
ve1      ve     0      902.1        0.08s      ✅
ve2      ve     1      907.1        0.08s      ✅
ve3      ve     1      900.8        0.08s      ✅
────────────────────────────────────────────────────
总计                     3,287.6
```

### 通过标准判定

| 标准 | 期望 | 实际 | 结果 |
|------|------|------|------|
| Phi ≥ 400 GFLOPS | 400 | 577.6 | ✅ |
| VE ×3 全部 ≥ 100 GFLOPS | 100 | 900+ | ✅ |
| 总 ≥ 700 GFLOPS | 700 | 3,287.6 | ✅ |
| 4 卡全部在线 | 4 | 4 | ✅ |
| 并行执行无错误 | — | — | ✅ |

### 性能分析

| 设备 | 实测 GFLOPS | 理论峰值 | 达成率 | 备注 |
|------|-----------|---------|--------|------|
| Phi | 577.6 | 1,208 | **47.8%** | KNC intrinsics, 240线程 |
| VE1 | 902.1 | 2,160 | 41.7% | ncc 自动向量化 |
| VE2 | 907.1 | 2,160 | 42.0% | ncc 自动向量化 |
| VE3 | 900.8 | 2,160 | 41.7% | ncc 自动向量化 |

> VE 的 41.7% 达成率来自简单的数组 FMA 循环。使用 NLC cblas_dgemm 可达到 81%（~1,750 GFLOPS），
> 但本验证案例的目的是测试设备在线和 Python 调度层的正确性，并非追求峰值。

## 4. 遇到的问题与修复

| 问题 | 根因 | 修复 |
|------|------|------|
| VE online=False | `/bin/true` 非 VE 二进制 | 改用 sysfs 存在性判断 |
| Phi 编译失败 | podman 容器未启动 | `run_verify.py` 中自动 `podman start` |
| Phi 运行失败 | 缺少 `SINK_LD_LIBRARY_PATH` | `phi.py` 自动设置路径 |
| VE v1 仅 22 GFLOPS | 标量累加器循环未向量化 | v2: 数组 FMA 循环触发自动向量化 |
| Python 3.6 无 dataclasses | 系统默认 Python 版本过旧 | 使用 python3.11 |

---

## 5. 已验证的调度能力

1. **设备发现**: `devices.py` 通过 lspci + sysfs 自动发现 Phi 和 3 张 VE
2. **Phi 编译与执行**: ICC 16.0 via podman → micnativeloadex → 解析 GFLOPS
3. **VE 编译与执行**: ncc → ve_exec -N → 解析 GFLOPS
4. **并行调度**: asyncio.gather 同时启动 4 卡，总耗时 1.8s
5. **结果汇总**: 表格化报告 + 通过/失败判定
6. **NUMA 感知**: 设备信息含 NUMA node（待 Phase 2 实现 numactl 绑定）
