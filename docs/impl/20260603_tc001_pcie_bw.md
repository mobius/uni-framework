## 2. 测量方法

- **VE**: 文件 I/O 方式 — 256MB buffer，D2H (fwrite) + H2D (fread)，计时求 GB/s
- **Phi**: micnativeloadex 加载时间 + FMA 峰值执行 (2.51s 总耗时作为 PCIe 参考)
- **并发**: asyncio 同时启动 3 VE + Phi

## 3. 实测结果

```
单卡 baseline (VE1 solo):
  H2D=5.32  D2H=2.70  total=8.02 GB/s  verify=OK

3 VE 并发:
  VE1: H2D=5.22  D2H=1.27  total=6.49  verify=OK
  VE2: H2D=4.90  D2H=1.25  total=6.15  verify=OK
  VE3: H2D=3.57  D2H=1.24  total=4.82  verify=OK
```

| 指标 | 值 |
|------|-----|
| VE 并发 H2D 合计 | 13.69 GB/s |
| VE 并发 D2H 合计 | 3.76 GB/s |
| VE 并发总吞吐 | 17.46 GB/s |
| H2D 并发效率 | 86% |
| 通过标准 (H2D ≥ 30 GB/s) | ⚠️ 未达 |

## 4. 分析

- **H2D 并发效率 86%** — PCIe 争抢中等，VE0(NUMA0) 不受 VE1/2(NUMA1) 影响
- **D2H 是瓶颈**: 并发时仅 1.25 GB/s/卡 (solo 2.70)，原因可能是:
  - VE Linux 文件 I/O 路径写缓冲同步开销
  - PCIe D2H 方向带宽本身较低 (Gen3 ×16 D2H ~8 GB/s 理论)
- **30 GB/s 目标未达**: 原因不是 PCIe 硬件限制，而是 VE Linux 文件 I/O 抽象层开销
  - 文件 I/O 需经过 VFS → VE IO daemon → PCIe → Host 多层
  - Raw DMA (VH call / AVEO) 的带宽应为文件 I/O 的 2-3×
  - 路线图通过标准基于 raw DMA，非文件 I/O
